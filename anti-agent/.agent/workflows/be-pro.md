# /be-pro — Backend Execution Engine

You are now activating the **Backend Execution Pipeline (BE-PRO)**. This is a rigorous, 3-gate pipeline designed to guarantee secure, scalable system development with REQ-ID traceability, Context Memory integration, and Double-Evidence reporting.

This workflow is strictly managed by the **Supervisor (Orchestrator)**.

---

## STRICT RULES OF ENGAGEMENT
1. **API First:** This pipeline defines the truth. Frontend MUST consume the contracts generated by BE-PRO.
2. **Context First:** Supervisor MUST query `.agent/.context/_index.json` before any schema or logic is generated.
3. **Double-Evidence:** Gate 3 requires BOTH script validation AND database/API response evidence.

---

## GATE 1: BLUEPRINT & TRACEABILITY
*Supervisor operates alone here.*

**Actions:**
1. **Short-Map Discovery**: Run `/graphify` or read `graphify-out/graph.json` to understand the system architecture and module dependencies.
2. **Semantic Search**: Use `mcp codebase-searcher` to identify all relevant controllers, services, and schemas.
3. Break down the user system request into logical backend features.
4. Assign each feature a strict Requirement ID (e.g., `REQ-BE-01: Auth Middleware`).
5. Define the Database Schema and the Output API Contract (JSON structure/Swagger).
6. Identify required Tools (e.g., Database MCP for schema checks, `curl` for API pinging).

**Gate Checkpoint:** 
- The Supervisor outputs the `BE Blueprint & Tool Manifest`.
- **STOP AND WAIT.** User must reply with "APPROVED" before moving to Gate 2.
- *If User provides feedback, revise Blueprint and loop.*

---

## GATE 2: CORE LOGIC EXECUTION
*Supervisor summons `database-architect` and `backend-specialist`.*

**Actions:**
1. **Context Injection:** Supervisor feeds the relevant Context Memory files (e.g., Prisma bugs, Hono.js configurations) to the Sub-agents.
2. **Schema & Code Generation:** Agents implement logic strictly adhering to the `REQ-ID` mapped in Gate 1.
3. **Skill Enforcement:** Must strictly follow `api-patterns`, `nodejs-best-practices`, or `python-patterns`. Avoid SQLi, trust no input.

**Mutex Lock Rule:**
- The Orchestrator locks the controller and schema files being edited. No other agent may write to these files until Gate 2 completes.

---

## GATE 3: 3-TIER QA & DATA PROOF
*Supervisor summons `security-auditor` and `test-engineer`.*

**Actions:**
1. **Tier 1 (Core):** Sub-agent verifies all `REQ-ID`s are fulfilled via unit tests.
2. **Tier 2 (Edge):** Check handling of null inputs, malformed JSON, missing tokens, and rate limits.
3. **Tier 3 (Perf/Sec):** `security-auditor` runs `vulnerability-scanner`. Ensure no hardcoded secrets, no SQLi.

**Double-Evidence Collection (Mandatory):**
1. **Script Evidence:** Save the console output of `security_scan.py` and `test_runner.py`.
2. **Data Evidence:** Run a command/script to query the API endpoint or Database locally and capture the raw response (e.g., `curl localhost/api/v1/health` or DB row dump). Save to Artifacts.

**Escalating Recovery (Debug):**
- If ANY Tier fails → Sub-agent attempts Self-Heal (Max 2 retries).
- If Self-Heal fails → Supervisor freezes Agent, delegates to `debugger` -> `debugger` fixes -> saves error to Context Memory -> resumes Pipeline.
